- QR 폐기 → 앱 내 진입 동선 재설계 필요
- 회원가입 시 가족 대리 가입 / 카카오 로그인 필요성 부상

### 실행 중 발생한 이슈
1. **`code --version`이 새 VSCode 창을 띄움**
   - 원인: PATH에 `Code.exe`(GUI)가 먼저 잡히고 `bin\code.cmd`(CLI)는 뒤
   - 해결: 사용자 PATH에 `bin` 폴더 추가 후 새 PowerShell
2. **`npm create vite@latest .` 시 빈 폴더 아님 경고**
   - 사용자 제안: 별도 폴더(`app/`)로 분리
   - 결정: `_readme/`(기획) + `app/`(소스 코드) 분리 구조
3. **Vite가 만든 `app/.gitignore`에 `.vscode/settings.json` 예외 없음**
   - 해결: 예외 라인 추가 후 다시 staged

### 학습 포인트
- "**클론한 사람이 따라 할 수 있는 문서**"는 PoC 자산 그 자체. 명령어, 결과 확인 방법, 트러블슈팅을 한 곳에.

생성 파일 19개:
- **공통 컴포넌트 3개**: `Button`, `ActivityHeader`, `CompleteButton`
- **활동 컴포넌트 4종**: `VideoActivity`, `BlankActivity`, `ColoringActivity`(초기 자유 그리기 버전), `DiaryActivity`
- **페이지 2개**: `HomePage`(진행률 + 7일 카드), `ActivityPage`(타입 분기 렌더링)
- **자산 1개**: `mandala-1.svg`
- **설정 정리 4개**: `App.tsx`, `index.css`, `App.css`, `tailwind.config.js`
- **문서 5개**: `00`(인덱스 업데이트), `21~24`(컴포넌트별 명세)

### 활동 컴포넌트 핵심 특징

#### 21. VideoActivity
- YouTube URL 파싱(`youtu.be/`, `watch?v=`, `embed/`) → ID 추출
- `react-youtube`로 16:9 비율 임베드
- 완료 버튼 → `progressStore.setDone()` → 0.8초 후 홈

#### 22. BlankActivity
- `template.split('{}')`로 빈칸 분리
- 입력 자동 저장 (localStorage)
- 정답 확인 → 빈칸별 ✅/❌, 카드 하단 정답 노출

#### 23. ColoringActivity (초기 버전)
- `react-konva` Stage + Layer + Line[]
- 마우스/터치 통합 이벤트
- 자동 저장(IndexedDB Blob, 800ms 디바운스)
- → 나중에 영역 클릭 방식으로 전면 전환

#### 24. DiaryActivity
- 4가지 섹션 종류 (radio / checkbox / text / textarea)
- 400ms 디바운스 자동 저장
- discriminated union으로 섹션별 분기 렌더

### 빌드/Lint 검증
- `npm run build`: ✅ 성공 (CSS 11kB, JS 577kB)
- `npm run lint`: ✅ 통과

### 산출물
- Git 커밋 `d746da2`: feat: implement 4 activity components for PoC

## 12. GitHub Pages 정적 배포

### 사용자 질문
> "혹시 깃헙 URL에서 이 앱을 실행할 수 있어?"

### Claude의 옵션 제시
| 방법 | 특징 |
|------|------|
| StackBlitz / CodeSandbox | 브라우저에서 즉시 실행 (개발 모드) |
| GitHub Codespaces | 클라우드 IDE |
| Vercel / Netlify | 정적 배포 + 자동 CD (가장 간편) |
| **GitHub Pages** | 정적 배포 + GitHub Actions 사용 |

### 사용자 선택
> "정적배포, Github Pages를 이용해서 하고 싶어. 조금 더 자세한 설명해줘"

### 사전 진단 (3가지 핵심 제약)
1. **코드가 `app/` 서브폴더** → 워크플로우에 `working-directory` 지정 필요
2. **`BrowserRouter` 사용 중** → GitHub Pages는 SPA fallback이 없어 `/path` 직접 접속이나 새로고침 시 404
3. **배포 URL이 `https://tank9567.github.io/haru_inji/` 형태** → 저장소 이름이 base path가 됨

### 3가지 수정

#### ① `app/vite.config.ts` — base path 지정
```ts
export default defineConfig({
  plugins: [react()],
  base: '/haru_inji/',
})
```
이유: 저장소 이름을 빌드 시점에 Vite에 알려줘야 모든 asset 경로(`/assets/xxx.js`)가 `/haru_inji/assets/xxx.js` 로 정상 생성됨.

#### ② `app/src/App.tsx` — Router 변경 (BrowserRouter → HashRouter)
- 옵션 A: HashRouter (URL이 `…/#/coloring` 형태, **설정 1줄**)
- 옵션 B: BrowserRouter + `public/404.html` SPA fallback 트릭 (URL 깔끔하지만 hacky)
- → **A 선택** (가장 안전·간단)

```tsx
import { HashRouter, Routes, Route } from 'react-router-dom';
// basename 불필요
```

#### ③ `.github/workflows/deploy.yml` — GitHub Actions 신규
핵심 포인트:
- `defaults.run.working-directory: app` — 모노레포 서브폴더 대응
- `cache-dependency-path: app/package-lock.json` — npm cache 키
- `upload-pages-artifact` path: `app/dist`
- `deploy-pages@v4` 로 GitHub Pages에 push
- `permissions: pages: write, id-token: write` 필수

### 사전 검증 (push 전)
- 로컬 `npm run build` 성공 (765ms, JS 267kB)
- `dist/index.html` 확인:
  ```html
  <link rel="icon" href="/haru_inji/favicon.svg" />
  <script src="/haru_inji/assets/index-…js"></script>
  ```
  → 모든 경로에 `/haru_inji/` prefix 정상 적용

### GitHub 저장소 설정 (한 번만)
- Settings → Pages → **Source: GitHub Actions** 로 변경
- 안 하면 build는 통과해도 deploy 단계에서 권한 오류로 실패

### 트러블슈팅

#### 이슈 1: `git add app/vite.config.ts` 실패
- 증상: `fatal: pathspec 'app/vite.config.ts' did not match any files`
- 원인: 사용자가 이미 `D:\_SRC\haru_inji\app` 폴더 **안에서** 명령 실행 → `app/app/...` 경로 중복
- 해결: `cd ..` 로 루트 이동 후 재실행 (또는 상대경로 `vite.config.ts src/App.tsx ../.github/workflows/deploy.yml`)

#### 이슈 2: Actions 페이지가 "Get started with GitHub Actions" 표시
- 증상: GitHub Actions 탭이 워크플로우 추천 화면을 보여줌
- 진단: `git status` 결과 — 파일은 modified지만 staged X, `.github/`는 untracked, push할 commit 없음
- 원인: 사용자가 push되지 않은 상태에서 Actions 페이지로 이동
- 해결: add → commit → push 후 페이지 새로고침 → 정상적으로 워크플로우 목록 표시

### 산출물
- Git 커밋 `2f918e2`: ci: deploy to GitHub Pages with HashRouter

### 학습 포인트
- **GitHub Pages는 "저장소명 = base path".** Vite의 `base` 옵션을 안 맞추면 흰 화면 + MIME type 에러.
- **SPA + GitHub Pages의 새로고침 404 문제**는 단골. HashRouter가 가장 빠른 회피책.
- **모노레포 워크플로우는 `defaults.run.working-directory` 하나로 깔끔하게.** 각 step에 `cd` 박을 필요 없음.
- **사전 검증(로컬 build → dist 결과물 inspect)**으로 CI 무한 재시도 비용 절감. 특히 `base` 같은 빌드 시점 옵션은 결과물에서 확인 가능.
- **CLI 경로 에러는 "지금 어느 디렉토리에 있는지" 확인부터.** 같은 명령도 cwd에 따라 다르게 동작.
- **"왜 안 보이지?" 류의 외부 시스템 문제는 로컬 상태부터 검증.** GitHub Actions 빈 화면 → push 안 됨 → commit 안 됨 → 결국 add부터 안 한 것.

## 14. 주요 학습 포인트

### 의사결정
1. **루트 결정부터.** 모든 결정을 동시에 다루지 말고 의존성 그래프로.
2. **PoC는 "안 만들 것"의 정의가 핵심.** 비범위(out of scope)를 문서에 박아두면 작업이 흔들리지 않음.
3. **사용자 시나리오가 플랫폼·UX·구현을 모두 결정.** "어르신 개인 스마트폰" 한 줄이 PWA/네이티브 선택을 좁힘.

### 문서화
4. **각 산출물에 단일 문서 하나.** 컴포넌트마다 별도 문서를 두면 변경 이력 추적·인수인계 쉬움.
5. **"00. 중요 정보" 같은 단일 진입점**이 온보딩 비용을 5분으로 줄임.
6. **"⚠️ 확인 필요"는 적극적으로.** 미해결 사항을 가시화해두면 잊혀지지 않음.

### 코딩
7. **데이터 모델은 콘텐츠 구조를 충실히 반영.** 단순화 위해 필드를 빼면 나중에 다시 추가하느라 더 큰 비용.
8. **명시적 요소 > 암묵적 CSS.** `<img>` 한 줄이 `background-image` 디버깅보다 빠름.
9. **번들 크기 감소가 곧 UX 개선.** 어르신은 저사양 단말 가능성 — react-konva 제거로 -54% 번들.
10. **기술적으로 어려운 길 ≠ 좋은 UX.** Canvas 자유 그리기보다 SVG 영역 클릭이 어르신에겐 적합.

### AI 협업
11. **자율 작업의 자유도는 사전 스펙에서 나온다.** 14번 PoC 스펙이 있었기에 "알아서 만들어놔"가 가능했음.

### 배포·인프라
14. **정적 배포는 base path와 라우터 모드 두 가지만 챙기면 90%.** 나머지는 CI 보일러플레이트.
