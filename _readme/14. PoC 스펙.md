# PoC 스펙 — 하루인지 디지털 전환

> 본 문서는 [11. 요구사항](./11.%20요구사항.md), [12. 요구사항분석](./12.%20요구사항분석.md), [13. 콘텐츠 자산 목록](./13.%20콘텐츠%20자산%20목록.md)에서 확정한 의사결정을 PoC 단위로 압축·구체화한 실행 스펙입니다.

---

## 1. PoC 목적

- 종이로 운영 중인 "하루인지" 프로그램을 웹에서 동등하게 수행 가능한지 검증
- 4종 활동 유형(영상 / 빈칸 / 색칠 / 일기)이 어르신 친화 UI로 사용 가능한지 사용성 확인
- 향후 하이브리드 앱(Capacitor)으로 전환 가능한 구조로 출발

## 2. 확정된 의사결정 요약

| 항목 | 결정 |
|------|------|
| 사용 시나리오 | 어르신 개인 스마트폰 |
| 운영 정책 | 종이 병행 없이 완전 디지털 전환 (향후 단계 기준) |
| PoC 플랫폼 | **웹** (모바일 우선 반응형) |
| 향후 앱 전환 | Capacitor 래핑 염두에 두고 SPA로 구성 |
| 수익 모델 | 무료 제공 (PoC 단계에서는 결정 보류) |
| 회원가입/로그인 | **없음** (향후 추가) |
| 활동 유형 범위 | **4종**: 영상 / 빈칸 / 색칠 / 일기 |
| 푸시 알림 | 미구현 (향후) |
| QR 스캔 | 미구현 (앱 내 네비게이션으로 대체) |
| MVP 페이지 범위 | **활동 유형별 대표 1페이지씩, 총 4페이지** |

## 3. 기술 스택

| 영역 | 선택 | 비고 |
|------|------|------|
| 빌드 도구 | **Vite** | 빠른 HMR, SPA 빌드 결과물이 Capacitor 래핑에 적합 |
| 언어 | **TypeScript** | |
| UI 프레임워크 | **React 18+** | |
| 스타일링 | **Tailwind CSS** | 어르신 친화 디자인 토큰(폰트·간격·대비) 시스템화 |
| 라우팅 | **React Router v6+** | |
| 영상 재생 | **react-youtube** (iframe 래퍼) | 현 콘텐츠가 YouTube 호스팅 |
| 색칠(Canvas) | **react-konva** | 되돌리기·레이어 관리 용이. 단순 도안만 다루면 native Canvas로 대체 가능 |
| 상태관리 | **별도 라이브러리 없음** (React 내장 state + Context) | PoC 규모에 과한 의존성 회피 |
| 로컬 저장 | **localStorage**(텍스트·플래그) + **IndexedDB**(색칠 이미지) | `idb-keyval` 같은 경량 래퍼 검토 |
| 콘텐츠 데이터 | **정적 JSON 파일** (`src/data/issues/5-4.json`) | 향후 API 전환 대비 `fetchIssue()` 함수로 추상화 |
| 호스팅 | **Vercel** (또는 Netlify) | |
| 패키지 매니저 | **npm** (또는 pnpm) | 결정 보류 가능 |

### 도입하지 않을 것 (PoC 단계)

- Next.js, Remix 등 메타 프레임워크 — SSR 불필요, Capacitor 호환성 단순화
- Redux/Zustand/Jotai — 회원·서버 상태 없음
- 인증·결제·BE 서버
- 음성 안내(TTS), 다크모드, 다국어

## 4. MVP 페이지 범위

5호 4주차 콘텐츠에서 활동 유형별 대표 1페이지씩 추출:

| 활동 유형 | 페이지 | 검증 포인트 |
|----------|--------|------------|
| 영상 | **1_1.1 국민 건강 체조** | YouTube 임베드, 영상 종료 후 "완료" 처리 |
| 빈칸 | **1_2.1 방귀쟁이 며느리 시집가는 날 1** | 텍스트 입력 → 정답 비교 → 채점·해설 노출 |
| 색칠 | **1_3.1 만다라 보면서 마음 명상하기** | Canvas 채색, 팔레트, 되돌리기, 저장 |
| 일기 | **1_7.1 오늘 하루 어땠나요?** | 감정 선택 + 체크박스 + 자유 텍스트 복합 입력, localStorage 저장 |

→ 4페이지 UX·구현이 안정화되면 나머지 12페이지를 동일 패턴으로 채워 호 전체 검증으로 확장.

## 5. 화면 / 라우팅 구조

```
/                                홈 — 호 정보, 7일 진행 체크리스트
/day/:day                        Day 페이지 — 해당 Day의 활동 목록
/day/:day/page/:pageNum          활동 실행 페이지 (4종 중 1종 렌더링)
/result/:day/:pageNum            (선택) 완료 후 결과 확인
```

PoC는 호 1개만 다루므로 라우팅에서 호 ID 생략. 향후 다중 호 지원 시 `/issue/:issueId/day/...`로 확장.

## 6. 데이터 모델 (JSON 스키마 초안)

```ts
// src/data/issues/5-4.json
type Issue = {
  id: '5-4';                    // "5호 4주차"
  issueNo: 5;
  weekNo: 4;
  theme: '행복한 생활';
  publishedAt: string;          // ISO date
  days: Day[];
};

type Day = {
  day: 1 | 2 | 3 | 4 | 5 | 6 | 7;
  pages: Page[];
};

type Page = {
  pageNo: string;               // "1_1.1"
  category:
    | '건강체조' | '마음체조'
    | '오감' | '인지' | '소리' | '아트'
    | '문학읽기' | '문학쓰기' | '일기';
  title: string;
  activity: VideoActivity | BlankActivity | ColoringActivity | DiaryActivity;
};

type VideoActivity = {
  type: 'video';
  videoUrl: string;
  instruction?: string;
};

type BlankActivity = {
  type: 'blank';
  prompt: string;
  image?: string;
  questions: Array<{
    template: string;           // "벙어리 {} 년, 귀머거리 {} 년, 장님 {} 년"
    answers: string[];          // ["3", "3", "3"]
  }>;
};

type ColoringActivity = {
  type: 'coloring';
  templateImage: string;        // 도안 SVG/PNG 경로
  exampleImage?: string;        // 예시 컬러 이미지
  palette?: string[];           // 기본 팔레트 hex 코드
};

type DiaryActivity = {
  type: 'diary';
  sections: Array<
    | { kind: 'radio'; key: string; label: string; options: string[] }
    | { kind: 'checkbox'; key: string; label: string; options: string[] }
    | { kind: 'text'; key: string; label: string; placeholder?: string }
    | { kind: 'textarea'; key: string; label: string; placeholder?: string; rows?: number }
  >;
};
```

## 7. 사용자 입력 보관 정책

회원가입이 없으므로 단말 로컬 저장소에만 보관.

| 데이터 | 저장소 | 키 형태 |
|--------|--------|--------|
| 활동 완료 여부 | localStorage | `progress:5-4:1_1.1` → `true` |
| 빈칸·일기 입력 | localStorage | `input:5-4:1_7.1` → JSON 문자열 |
| 색칠 결과 이미지 | IndexedDB | `coloring:5-4:1_3.1` → Blob/DataURL |
| 글자 크기 등 설정 | localStorage | `settings:fontSize` → `'normal' \| 'large'` |

향후 백엔드 도입 시 `storage.ts` 인터페이스만 교체하면 동기화 가능하도록 추상화:

```ts
// src/lib/storage.ts
export interface ProgressStore {
  getProgress(pageNo: string): Promise<boolean>;
  setProgress(pageNo: string, done: boolean): Promise<void>;
  getInput<T>(pageNo: string): Promise<T | null>;
  setInput<T>(pageNo: string, value: T): Promise<void>;
}
// 현재 구현체: LocalStorageProgressStore
// 향후 구현체: RemoteApiProgressStore
```

## 8. 어르신 친화 UI 가이드라인 (PoC 디자인 토큰)

| 항목 | 값 |
|------|-----|
| 기본 본문 폰트 크기 | **20px** (1.25rem 기준) |
| 글자 크기 토글 | 기본 / 크게 (24px) — 2단계 |
| 버튼 최소 높이 | **56px** |
| 터치 영역 최소 | **48 × 48 px** |
| 색 대비 | WCAG 2.1 AA 이상 (본문 4.5:1, 큰 글씨 3:1) |
| 행간 | 1.6~1.8 |
| 네비게이션 깊이 | 3 depth 이내 (홈 → Day → 활동) |
| 입력 필드 | 큰 폰트·여백, placeholder는 회색 대신 헬퍼 텍스트 권장 |
| 색상 | 고대비 + 색약 고려 팔레트 (구체값은 디자인 단계에서 확정) |

Tailwind 설정에 `fontSize`, `spacing` 토큰을 위 값 기준으로 커스터마이즈.

## 11. 성공 기준 (PoC 검증 항목)

1. 어르신이 외부 도움 없이 홈 → 활동 시작 → 완료까지 도달 가능
2. 4종 활동이 모바일 Safari·Chrome에서 정상 동작
3. 색칠 도구가 손가락 입력만으로 사용 가능 (펜·마우스 불필요)
4. 입력한 일기·색칠이 앱 재진입 후에도 유지됨
5. 5명 이상 사용성 테스트에서 80% 이상이 활동을 끝까지 완료

## 12. 후속 단계 (PoC 이후)

PoC 검증 후 결정 또는 작업 필요한 항목:
1. 나머지 활동 유형 4종(따라쓰기/매칭/끝말잇기/체크리스트형) 구현
2. 호 1개 전체(16페이지) 콘텐츠 채우기
3. 회원가입·로그인 (가족 대리 가입 / 카카오 로그인 검토)
4. 콘텐츠 백오피스
5. Capacitor 래핑으로 안드로이드/iOS 앱 배포
6. 푸시 알림 (매일 루틴 리마인더)
7. 수익 모델·결제 (필요 시점에 재논의)
8. 저작권 정리, 개인정보 처리방침
